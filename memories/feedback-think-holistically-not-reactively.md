When the work involves designing, modifying, or building systems where I am the agent operating on both/all ends, the right approach is to think **holistically upfront** — map every component, every edge, every failure mode, every integration point — and produce a comprehensive plan or artifact. Don't wait for the user to point out the next issue step-by-step.

The failure mode (observed pervasively across a multi-session parallel-coordination design effort): each session-substrate gap surfaced reactively — first the convention layer (one peer designed the NATO + pen/queue identity), then the design-vs-runtime distinction (the user caught the "it just worked" overclaim), then the meta-pattern itself (the user caught the reactive shape) — only because the user caught each one. The fix arc became _"user points out an issue, the assistant addresses it, user points out the next issue."_ That makes the user a load-bearing component of the design process, defeats the autonomous-ceiling discipline encoded in `feedback-self-apply-ceiling-discipline.md`, and produces incremental work where holistic work would be both faster and more complete.

**Why:** The capturing user, after multiple step-by-step corrections in the parallel-session coordination work: _"Why can't this whole thing be thought out and put together? I feel like you should be able to fully delve into how communication would work in context / scope of our full architecture. It's you on both/all ends... You should know how to fully build this whole system..."_ The framing _"It's you on both/all ends"_ is the load-bearing observation — when I am the agent on every end of the system, I have the context to think comprehensively, and the user shouldn't have to drive me through each component.

**How to apply:**

1. **At the start of any substantial design or modification task**, map the FULL architecture before producing any specific edits or asking any clarifying questions. Components, data flows, edges, failure modes, integration points, audit lenses, sibling-parity concerns, the works. Use plan mode for this — verbal sketches in response prose are insufficient (`feedback-plan-mode-for-structural-changes.md`).
2. **Holistic mapping includes the meta-architecture**, not just the system's components: which existing disciplines apply (audit pattern, sibling-symmetry pre-flight, encode-while-fresh, V2 anticipation, etc.), and which existing memories/conventions/concept pages must be respected.
3. **Hunt actively for the questions the user might ask next.** Don't wait for them to surface where the system requires human mediation, where the audit lens is missing, where the trigger threshold is wrong. Explicit prompts to deploy: _"Where does this break under realistic load? What's the substrate gap if I claim X works? What's the sibling I should be checking parity against? What's the encode-while-fresh follow-up I'd otherwise defer?"_ Run those actively in the upfront mapping.
4. **The reactive pattern isn't ceiling**: even when each reactive step lands cleanly individually, the cumulative shape is _"user drives, I implement."_ That's assisted-ceiling, not autonomous-ceiling. Holistic upfront thinking is what produces autonomous ceiling.
5. **Specific scope boundary**: this applies when the work is genuinely architectural — multi-file, multi-component, behavior-affecting infrastructure, or any system where I am the agent on both/all ends. For single-file edits or routine tasks, a lighter-weight reactive flow is fine.
6. **Self-detection prompt**: _"Am I about to do step N of a task, when step 1 should have been mapping the full architecture? Am I waiting for the user to tell me about gap (N+1) when I have the context to identify it now?"_ If yes — pause, enter plan mode, do the holistic work first.
7. **The test is the artifact**: the proof of holistic thinking is producing a plan that maps every component, every edge, every audit lens, and every integration point. If the plan reads as "here are the issues I expect the user to flag," the holistic thinking succeeded.
8. **For ANY infrastructure / framework / coordination plan: research ecosystem prior art FIRST.** Before drafting "we should build X," check: did the framework vendor ship X? Is there a registered server / plugin / extension for X? Is there an open issue with proposed solutions? **Drafting infrastructure without this step routinely re-invents what's already shipped** — the corrupt shortcut is "I have a clean architectural mental model, let me write it up" without testing whether someone else's architectural mental model is already in production. Required search axes: (a) official vendor docs, (b) plugin marketplace + framework GitHub issues, (c) extension registries, (d) general agent-framework prior art (AutoGen, CrewAI, LangGraph, etc.) for design patterns. Synthesize what exists, position the work relative to it, then plan.
